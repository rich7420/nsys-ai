Performance Questions That Matter (Including MFU)

A curated set of high-value questions for GPU training performance analysis. This is the shortlist to ask first, before deep-diving into dozens of secondary checks.

1) The Top Questions To Ask First

Priority Question Metric To Compute Typical Action If Bad
P0 What is MFU right now, and what is its trend? MFU = achieved_model_TFLOPS / peak_TFLOPS Fix biggest non-compute bottleneck before kernel micro-optimizations.
P0 Where does step time go (compute vs comm vs idle)? Step-time decomposition by category Attack the dominant bucket first (NCCL, bubbles, or compute kernel).
P0 Is communication overlapping compute? NCCL overlap ratio + NCCL share of step Async comm, stream strategy, bucket sizing, topology checks.
P1 Are there major GPU bubbles (idle gaps)? Idle-gap ms and % of step span Pipeline scheduling, input staging, synchronization cleanup.
P1 Which kernels dominate runtime? Top-kernel cumulative % Kernel-level optimization/fusion/algorithm choice.
P1 Are we losing time to many tiny kernels? Launch overhead + tiny-kernel count Fuse kernels, raise arithmetic intensity, reduce dispatch count.
P2 Are H2D/D2H transfers in the critical path? Transfer ms by direction + timeline placement Prefetching, pinned memory, remove accidental host syncs.
P2 Are ranks/GPUs imbalanced? Per-rank step time distribution (p95/p50) Data sharding, sequence packing, straggler root-cause analysis.
P2 Is first iteration an outlier? Warmup step vs steady-state step Move compilation/init outside measured window.
P3 Did MFU improve after change X? Before/after MFU + confidence interval Keep only wins that survive repeated profiles.

2) MFU: What It Is And How To Calculate It

MFU (Model FLOP Utilization) answers: how much of theoretical GPU compute peak is being converted into model math.

Step A: Model FLOPs per step Estimate from model config and tokens/step.
Step B: Achieved model TFLOPS model_flops_per_step / step_time_s / 1e12
Step C: MFU achieved_model_TFLOPS / peak_TFLOPS_per_gpu

Core Formula

tokens_per_step = global_batch_size * sequence_length

model_flops_per_step = flops_per_token * tokens_per_step

achieved_model_tflops = (model_flops_per_step / step_time_s) / 1e12

MFU = achieved_model_tflops / peak_tflops_per_gpu

Practical FLOPs-per-token Baseline

For dense Transformer training, a common rough baseline is: 

flops_per_token ~ 6 * N_parameters

This is a simplification. Real workloads need corrections for attention, MoE routing, activation recompute, sequence parallelism, and optimizer variants.

Nsight Systems alone does not directly provide true model FLOPs. It provides timing and kernel traces. MFU needs model-side FLOP accounting inputs in addition to timeline data.

Cluster / Multi-GPU Variant

cluster_MFU =
  total_model_flops_per_step
  /
  (num_gpus * peak_tflops_per_gpu * step_time_s)

MFU vs HFU (Do Not Mix Them)

MetricNumeratorNeedsUse Case
MFU Model FLOPs/s (algorithmic math) Model FLOP accounting + step time + peak TFLOPS Training-system efficiency at the model level
HFU Hardware FLOPs/s (instructions executed) Hardware counters (typically Nsight Compute) Kernel/hardware efficiency of executed instructions

4) Minimal SQL Snippets For The Key Questions

Top kernels by total duration

SELECT s.value AS kernel_name,
       COUNT(*) AS invocations,
       ROUND(SUM(k.[end] - k.start) / 1e6, 2) AS total_ms
FROM CUPTI_ACTIVITY_KIND_KERNEL k
JOIN StringIds s ON k.demangledName = s.id
GROUP BY s.value
ORDER BY total_ms DESC
LIMIT 15;

NCCL cost breakdown

SELECT s.value AS kernel_name,
       COUNT(*) AS n,
       ROUND(SUM(k.[end] - k.start) / 1e6, 2) AS total_ms
FROM CUPTI_ACTIVITY_KIND_KERNEL k
JOIN StringIds s ON k.shortName = s.id
WHERE s.value LIKE '%nccl%' OR s.value LIKE '%NCCL%'
GROUP BY s.value
ORDER BY total_ms DESC;

Idle gaps (pipeline bubbles)

WITH ordered AS (
  SELECT k.streamId, k.start, k.[end],
         LAG(k.[end]) OVER (PARTITION BY k.streamId ORDER BY k.start) AS prev_end
  FROM CUPTI_ACTIVITY_KIND_KERNEL k
)
SELECT streamId,
       ROUND((start - prev_end) / 1e6, 3) AS gap_ms
FROM ordered
WHERE prev_end IS NOT NULL AND (start - prev_end) > 1000000
ORDER BY gap_ms DESC
LIMIT 20;

These answer decomposition questions that are prerequisites for interpreting MFU correctly.

5) A Practical Workflow

  1. Measure step time and decomposition (compute/comm/idle).
  2. Compute MFU with explicit FLOPs-per-step assumptions.
  3. State assumptions in report (precision, token count, FLOP model).
  4. Fix the largest bottleneck category.
  5. Re-profile and report MFU before/after with same workload.
